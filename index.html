202701杂记

2021年七月，夏，天气阴霾

火车从长沙开到了西安，我也随火车回到了西安这个生活了五年的地方。

七月，全国的天气好像都似着了火一样，但也有不同，长沙的热能让人汗如雨下，豆大的汗水顺着脸颊可以很轻易地滑落到公交车的地板上；而西安的热没有那么多水的成分，走在街上身上的汗也没有出很多。
这次回来感觉西安这座城空荡荡的，也许是因为她走了，这是我第一次在西安逗留而没有见到她。约了同学吃饭，只是因为想要再回家之前可以吃到好久没吃的烤肉，见面之后好像我的话相较于之前多了好多，在长沙的这些日子基本上我也没有什么话可以对周围的人去讲，每天周而复始干着同样的工作、但每天又有着些许不同，但总归没有什么太大的变动。
这是一篇杂记，但我又想写一些总结性质的东西。回到家，意味着研一的彻底结束，在对导师阳奉阴违的做派下我度过了这一年，我很清楚我想要的未来，我也觉得我有为之努力，但总觉得还差很多很多，距明年春招只剩七个月，因此需要做一些更加具体而又实际的东西来填补我在实际项目上的经验空缺，但论文还是没有头绪，虽然使用Python去学习、复现论文中的模型基本上没有什么问题，但总结和归纳自己的结果却有些没有头绪，所以接下来的几个月方向性要更加明确一些，虽然不喜欢科研这种养蛊的东西、但是毕业条件还是得拿到不然会让自己处在一种很尴尬的位置之上。而关于工作之事也到了要决战冲刺的时候了。
我也不知道明年能是一个怎么样的结果，我渴望着能够进入大厂，渴望着能够去往业界一流的公司去工作，但是我的水平真的足以支撑我的野心吗？还是说我的一切都只是一个笑话。

隐约雷鸣、阴霾天空

前途好像覆盖着厚重的迷雾、我不知道到底出路该如何去走。她走了，好像从未来过，只留下了很多我们之间的曾经相处的痕迹。生活中总是带着她的影子，这一次她应该是真的走了。我也不明白为什么我还是不愿放弃，可能是不愿放弃之前的自己的付出，也可能是不愿和她美好的回忆就此画上终结符。但更加浓重的阴霾是学业和工作带来的，学业的懈怠和工作的不确定性所带来的内心踌躇总会一遍遍地折磨着我。这也许就是知乎里那些人所说的精神内耗。我总是借口找工作地学习来遮蔽掉对论文的研究以及加快自己看新东西的速度而不去练习，这使得我所学并不牢固，而且由于看的书还都是大部头，所以知识的遗忘总比学得要快，而这种情况更加加深了自己内心的焦虑。之后的策略要发生改变、要让自己的目标更加明确、要更加专注地去解决问题。

但盼风雨来

读研的日子总有两种截然相反的想法在脑中对立。希望能够给我更多的时间去磨练技术，沉淀技术。以及能够过得快一点，这穷苦的日子，导师对论文的催促以及一个人的孤独总是使得我希望能够将钟表直接拨到毕业的日子，让我的生活能够发生一些重大的变动，改变我现有的生活。我不喜欢研究生的生活、我也不喜欢每天看论文的日子（虽然我也没有每天去看，但是那种负罪感还是能够将我压的有些沉闷），总之我不希望过平静的生活，但总受迫于没钱和菜，使得我的生活只得一直沉闷下去。

能留你于此地

我总是会去权衡，脑子里总有一些不切实际的想法，但最终我明白，我可能什么也不会得到，我的一生会是一直在惋惜的一生。吗？

对给定的字符串s, 长度为n, 其中的字符从左到右依次为s[1], s[2], …, s[n]。使用动态规划的方法计算出字符串s的解码方法数。

设fi表示字符串s的前i个字符s[1..i]的解码方法数。在状态转移时，考虑最后一次解码使用了s中的哪些字符，会有如下情况：

  • 第一种情况时我们使用了一个字符，即s[i]进行解码，那么只要s[i]!=0,其就可以被解码成A~Izhong的某个字母。 由于剩余的前i-1个字符的解码方法数为$f{i-1}$，因此我们有状态转移方程:$f_i = f{i-1}$其中s[i] != 0
  • 第二种情况是我们使用了两个字符，即s[i-1]和s[i]进行编码。其与第一种情况类似，s[i-1]不能等于0，并且s[i-1]和s[i]组成的整数必须小于等于26，这样他们就能被解码为J~Z中的某个字母。由于剩余的前i-2个字符的解码方法数为$f{i-2}$状态转移方程为：$f_i = f{i-2}$其中s[i-1]!=0 且10 * s[i-1] + s[i] <= 26;

将上述两种情况进行累加即可得到最终的$f_i$值。

边界条件$f_0=1$。

最原始的DIP定义为：

这句话也是，重申了对接口或基类的依赖比对具体类型的依赖要好。

Trie前缀树或字典树，是一棵有根树，节点字段如下：

  • bool字段isEnd表示该节点是否为字符串的结尾

向树中插入字符串

从字典树的树根开始，插入字符串。对于当前字符对应的子节点，有两种情况：

  • 子节点存在。沿着指针移动到子节点，继续处理下一字符。
  • 子节点不存在。创建一个新的子节点，记录在children数组的对应位置上，然后沿着指针移动到子节点，继续搜索下一字符。

重复以上步骤，直到处理字符串的最后一个字符，然后当前节点标记为字符串的结尾。

查找前缀

从字典树跟开始查找前缀。对于当前字符对应的子节点，有两种情况:

  • 子节点存在。沿着指针移动到子节点，继续搜索下一个字符。
  • 子节点不存在。说明字典树中不包含该前缀，返回空指针。

重复以上步骤，知道返回空指针或搜索完前缀的最后一个字符。

若搜索到了前缀的末尾，就说明字典树中存在该前缀。此外，若前缀末尾对应节点的isEnd为真，则说明字典树中存在该字符串。

四次挥手：

  1. 主动关闭这发送FIN，包含一个序列号K和最后一个接收报文L-1的ACK值为L。
  2. 被动关闭者将K+1作为ACK返回，报文序号为L。
  3. 被动关闭者发送自身FIN+ACK， 报文序号为L，ACK为K+1（和上面一样）。
  4. 主动关闭者发送ACK序号为K，ACK为L+1

TCP半关闭-仅关闭数据流的一个传输方向，而两个半关闭操作合在一起就能关闭整个连接

半关闭

TCP主动关闭端指出己端数据发送完毕，但仍希望接收对端数据。

过程如下：

  1. 最先的两个报文段与先前一直。
  2. 在被动关闭端发送完FIN的确认后，仍可发送数据。
  3. 数据发送结束，被动关闭端发送FIN来完成关闭。
同时打开与关闭

两端SYN几乎同时发送到达对端，双方均拥有IP地址和端口号，在初始的SYN到达两端后，两端分别就这两个SYN进行确认，之后连接建立。此种情况下确认序列号使用SYN序列号，也就是序列号仍保持原状，将对端序列号加一作为ACK返回。

两端同时发送FIN，在收到FIN后两端又同时确认，序号不变，将对端序列号加一作为ACK返回。

初始化序列号

发送SYN前随机选择选择序列号，避免序列号重叠问题（可能的实现32位计数器，4us加一改变ISN）。

同一连接的不同实例，关闭后又建立的不同实例，采用相同四元组导致可能的后续数据延迟发送到之前的连接。因此为了避免这种情况，初始化序列号的选择极为重要。

随机选择也是为了抵御攻击，因为只要一个TCP报文段具备连接的四元组和活动窗口的序列号即会被认为是正确的，因此为了不让攻击者猜出序列号，须随机。

分组窗口和滑动窗口

分组窗口：已发送但未完成确认的分组集合—其中的分组数量称之为窗口大小。

窗口前部的分组被确认，分组进行滑动，此时分组中包含已发送未确认和一个新的未发送但马上发送的分组，拥有这种操作所以称之为滑动窗口。

滑动窗口-发送方：可释放分组、等待ACK分组以及未发送分组。

滑动窗口-接收方：已接受分组、已确认分组、期望分组及内存限制被丢弃分组。

变量窗口：流量控制和拥塞控制

流量控制

发送接收方速度不对等。

基于速率：发送发指定某个速率，确保数据发送速率不超过此速率

基于窗口：窗口大小随时间变动-通知接收方窗口通告。发送方边滑动边改边大小。

接收方窗口通告夹杂发送方窗口大小时，发送方速率被限制不能超越接收方。

拥塞控制

使分组发送速度降低以适应链路状态。
